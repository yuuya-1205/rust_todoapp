# コーディングルール

このプロジェクトでコードを書くときに従うルールをまとめます。
基本方針は `AGENTS.md` に、モデルファイル固有の記述規約は `src/model/model.md`
（`rust-model-doc` スキル）にあります。ここでは全体に共通するルールを補足します。

## 1. 言語・コメント

- コメントは**日本語**で書きます。既存の日本語コメントを勝手に英語へ翻訳しません。
- 学習目的のプロジェクトのため、「なぜそうするのか」が伝わる手厚いコメントを尊重します。
- 公開ドキュメントコメントは `///`、モジュール全体の説明は `//!` を使います
  （詳細は `src/model/model.md`）。

## 2. モジュール / 公開範囲

- ディレクトリをモジュールとして扱う場合、入口ファイルは `mod.rs` にします。
- `mod.rs` では、そのディレクトリ配下で公開するモジュールだけを `pub mod` で宣言します。
- 公開は最小限にします。外部から使う必要がないものに `pub` を付けません。
- `main.rs` には詳細な処理を書かず、各レイヤーの入口を呼び出すだけにします。

## 3. レイヤー分割

- `model/` … データ構造（ドメイン）。アプリが扱う「もの」を表す構造体を置きます。
- `presentation/` … ユーザーとの接点（CLI）。入力・表示・操作フローを置きます。
- レイヤーをまたぐ依存は、入口（`mod.rs`）経由で最小限にします。
- 詳細な設計意図は `docs/architectured.md` を参照してください。

## 4. 命名

- ファイル名・モジュール名は snake_case（例: `todo.rs`, `model`）。
- 構造体・列挙型は UpperCamelCase（例: `Todo`）。
- 関数・変数・フィールドは snake_case（例: `input_todo`, `title`）。

## 5. フォーマット / Lint

- コミット前に `cargo fmt` で整形します。
- `cargo clippy` の警告は可能な限り解消します。

## 6. 検証

- コードを変更したら、最低でも `cargo build` と `cargo test` が成功することを確認します。
